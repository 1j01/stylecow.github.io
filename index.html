Why another CSS processor?

The main purpose of stylecow is support modern CSS in all browsers. Stylecow transform your code to be compatible with all browsers adding vendor prefixes, fixing issues, generating polyfills, etc.

You can use latests (or futures) css features such variables, custom media, color manipulation, etc using the w3c syntax. So, you don't have to learn a different syntax as with preprocessors such LESS or SASS to use these features. Stylecow is a postprocessor, it means it works with pure css. You write css and you get css, and if you stop using stylecow, your css code remains css code.

What about postcss?

Postcss is a popular and awesome css postprocessor, very similar to stylecow but it has some issues that stylecow tries to solve:

Order matters

In postcss, the order of the plugins you are using is important because they are executed one at a time, so if you want combine several plugins, this can be a problem. For example, if you define postcss-css-variables before postcss-import, the first plugin wont be executed in the files imported by the second one. And there are some recursive cases very hard to solve, for example, variables using color functions using variables. Cssnext tries to solve this (and do it pretty well), but if you want combine it with more plugins, you can have the same problem.

Create plugins is not easy

Postcss parser is very, very fast (one of the fastest). But this is because its AST doesn't go so far, and it's not easy to work with little things like numbers, units, etc.

Let's take, for example the postcss-vmin plugin. To search all vmin units in the css code it need regular expressions, .indexOf(), etc. The stylecow version of this plugin is cleaner. It's API allows work with css in a safer way, so if you have special needs, you can manipulate the css code easier.

Better syntax checking

Postcss detects some simple css syntax errors, like unclosed rules or functions. But issues like a[href= or display: .none are not detected and are considered valid. Stylecow validates the code and can detect more syntax errors.

Code output

Stylecow allows to customize the output code generation. There are two styles by default (normal an minify) but more styles can be created easily

Note

I don't have anything against postcss, I take it as example because it is one of the most popular library with a similar purpose than stylecow. It's a great option, better than stylecow some aspects. I only want to expose some parts that I think stylecow is better.

Quick start

Use npm to install the library. Add -g if you want install it globally:

npm install stylecow -g

Now you have the stylecow command with some useful utilities. The next step is create a file with the configuration. You can create it in an interactive way using the command:

stylecow init

In this process, you have to define the following settings:

  • support: Minimum version supported by the browsers explorer, firefox, chrome, safari, opera, android and ios
  • plugins: The plugins you want to use
  • code: Code style used to ouput the css. (normal and minify)
  • input: path of the main css file to convert
  • output: path of the file where save the result
  • map: whether or not generate the source map

A typical config file looks like:

{
    "files": [
        {
            "input": "styles.css",
            "output": "styles.min.css",
            "map": "styles.min.map"
        }
    ],
    "support": {
        "explorer": 8,
        "firefox": 30,
        "chrome": 35,
        "safari": 5,
        "opera": 22,
        "android": 4,
        "ios": 6
    },
    "plugins": [
        "color",
        "variables",
        "nested-rules"
    ],
    "code": "minify"
}

By default the configuration file is saved as stylecow.json but you can choose a different name. To run this configuration, just execute:

stylecow

You can use the following options:

  • -c, --config <file> Change the default config file (./stylecow.json)
  • -w, --watch To keep watching the changes in the input files and generate the output automatically
  • -r, --live-reload Live reload the css in the browser when the output changes (using a websocket)

Using with Grunt/Gulp

Are you one of these guys? No problem, there is a grunt plugin available that you can install with npm:

npm install grunt-stylecow --save-dev

Now, you can register and configurate it in the Grunt file:

module.exports = function(grunt) {
    grunt.initConfig({
        stylecow: {
            options: require('./stylecow.json'), //Your config file
            dist: {
                files: {
                    'css/styles.min.css': ['./css/styles.css']
                }
            }
        }
    });

    grunt.loadNpmTasks('grunt-stylecow');

    grunt.registerTask('default', ['stylecow']);
}

Or if you are a Gulp-guy, here is your plugin: 

var gulp = require('gulp');
var stylecow = require('gulp-stylecow');

gulp.task('stylecow', function() {
  gulp.src('./css/styles.css')
    .pipe(stylecow(require('./stylecow.json')))
    .pipe(gulp.dest('./dist/'))
});

Plugins

All stylecow operations are done via plugins. There are several plugins available, for example, to play with variables using the w3c syntax, to manipulate the colors using the CSS Colors Module Level 4 syntax, to play with the latest syntax of flexbox, etc... All these plugins are included in your stylecow installation, so no extra installations are required.

The available plugins are the following:

Support for CSS incoming features:

color: manipulate colors using the CSS Colors Module Level 4 syntax
matches: add support for :matches() selector
calc: add support for calc() function (only if no more than one unit is used)
custom-media: add support for @custom-media at-rule
custom-selector: add support for @custom-selector at-rule
variables: allows the use of CSS variables

Language extensions:

nested-rules: unwraps nested rules
extend: use placeholders and @extend

Utilities:

import: To insert the code of the @import rules with relative urls

Code fixing:

fixes: Fixes automatically some well-know browsers issues
flex: To use the latest syntax of flexbox, generating the fallback for browsers supporting deprecated syntax.
prefixes Adds automatically all vendor prefixes needed
rem Adds fallback for rem values in pixels

Emulation of some features in old Internet Explorer using ms-filters:

msfilter-background-alpha Emulate the rgba() background in Internet Explorer 8 and lower using ms filters
msfilter-linear-gradient Emulate the linear-gradient() background in Internet Explorer 8 and lower using ms filters
msfilter-transform Emulate the CSS 2d transformations in Internet Explorer 8 and lower using ms filters
